# ServiceFlow - Client & Service Management SaaS

A complete production-ready SaaS platform for managing clients and services with a modern, clean architecture.

## ЁЯЪА Features

- **Authentication & Authorization**
  - JWT-based access tokens (15-minute expiry)
  - Refresh tokens (7-day expiry) stored in database
  - Role-based access control (Admin, Manager, User)
  - Secure password hashing with bcrypt

- **Client Management**
  - Create, read, update, delete clients
  - Search and pagination
  - Email validation
  - Audit logs for all changes

- **Service Management**
  - Create, read, update, delete services
  - Service status tracking (pending, in progress, completed)
  - Service value tracking for revenue calculation
  - Filter by client and status
  - Server-side pagination

- **Dashboard & Metrics**
  - Total clients count
  - Total services count
  - Services by status breakdown
  - Total revenue calculation
  - Real-time metrics refresh

- **Comprehensive Audit Logging**
  - Track all CRUD operations
  - User action logging
  - Old and new values comparison
  - Query by entity or user

### Role-Based Permissions

## ЁЯЪА Quick Start

### Prerequisites
- Docker & Docker Compose (recommended)
- Node.js 18+ (for local development)
- PostgreSQL 15+ (if running locally)

### With Docker (Recommended)

1. **Clone and setup**:
```bash
cd ServiceFlow
chmod +x setup.sh
./setup.sh
```

2. **Access the application**:
- Frontend: http://localhost:5173
- Backend API: http://localhost:3000
- PostgreSQL: localhost:5432

3. **Default Credentials**:
```
Admin Account:
- Email: admin@serviceflow.com
- Password: admin123

Manager Account:
- Email: manager@serviceflow.com
- Password: manager123

User Account:
- Email: user@serviceflow.com
- Password: user123
```

4. **View logs**:
```bash
docker-compose logs -f backend
docker-compose logs -f frontend
```

5. **Stop services**:
```bash
docker-compose down
```

### Local Development

#### Backend Setup

1. **Install dependencies**:
```bash
cd backend
npm install
```

2. **Setup environment**:
```bash
cp .env.example .env
```

3. **Configure database** (update .env):
```
DATABASE_URL=postgresql://user:password@localhost:5432/serviceflow
```

4. **Run migrations**:
```bash
npm run db:migrate
```

5. **Seed database**:
```bash
npm run db:seed
```

6. **Start development server**:
```bash
npm run dev
```

Backend runs on: http://localhost:3000

#### Frontend Setup

1. **Install dependencies**:
```bash
cd frontend
npm install
```

2. **Setup environment**:
```bash
cp .env.example .env
```

3. **Start development server**:
```bash
npm run dev
```

Frontend runs on: http://localhost:5173

## я┐╜ Deployment

### Recommended Platforms

#### 1. Render (Easiest - Free Tier Available)
1. **Create Render Account** at [render.com](https://render.com)
2. **Connect your GitHub repository**
3. **Use the blueprint**: The `render.yaml` file will automatically configure:
   - PostgreSQL database
   - Backend API service
   - Frontend static site
4. **Set environment variables** (optional - Render generates secure ones):
   - `JWT_SECRET`: Auto-generated by Render
   - `DATABASE_URL`: Auto-provided by Render PostgreSQL
5. **Deploy**: Click "Apply" and wait for deployment

#### 2. Railway (Great for Full-Stack)
1. **Create Railway Account** at [railway.app](https://railway.app)
2. **Connect your GitHub repository**
3. **Railway will detect** the `railway.toml` configuration
4. **Services created**:
   - PostgreSQL database (automatic)
   - Backend service
   - Frontend service
5. **Environment variables** are auto-configured

#### 3. Vercel + Railway (Frontend on Vercel, Backend on Railway)
1. **Deploy backend** to Railway (see step 2 above)
2. **Deploy frontend** to Vercel:
   - Connect GitHub repo
   - Set `VITE_API_URL` to your Railway backend URL
   - Vercel detects the `vercel.json` config

#### 4. Docker Production Deployment
```bash
# Setup production environment
cp .env.prod.example .env.prod
# Edit .env.prod with your production values

# Deploy with Docker Compose
docker-compose -f docker-compose.prod.yml up --build -d

# Or use the deployment script
chmod +x deploy.sh
./deploy.sh production
```

```http
{
```


#### Logout
```http
POST /auth/logout
Content-Type: application/json

{
  "refreshToken": "refresh_token"
}
```

### Clients Endpoints

#### Create Client (Admin/Manager only)
```http
POST /clients
Authorization: Bearer {accessToken}
Content-Type: application/json

{
  "name": "Client Name",
  "email": "client@example.com"
}
```

#### List Clients
```http
GET /clients?page=1&pageSize=10&search=name
Authorization: Bearer {accessToken}
```

#### Get Client
```http
GET /clients/{id}
Authorization: Bearer {accessToken}
```

#### Update Client (Admin/Manager only)
```http
PUT /clients/{id}
Authorization: Bearer {accessToken}
Content-Type: application/json

{
  "name": "Updated Name",
  "email": "updated@example.com"
}
```

#### Delete Client (Admin only)
```http
DELETE /clients/{id}
Authorization: Bearer {accessToken}
```

## ЁЯУК Database Schema

### Users Table
- id (UUID)
- email (String, unique)
- password (String, hashed)
- role (Enum: admin, manager, user)
- createdAt (DateTime)
- updatedAt (DateTime)

### Clients Table
- id (UUID)
- name (String)
- email (String, unique)
- createdAt (DateTime)
- updatedAt (DateTime)

### Services Table
- id (UUID)
- clientId (FK to Clients)
- title (String)
- description (String)
- value (Float)
- status (Enum: pending, in_progress, completed)
- createdAt (DateTime)
- updatedAt (DateTime)

### Audit Logs Table
- id (UUID)
- userId (FK to Users)
- entityType (String)
- entityId (String)
- action (Enum: CREATE, UPDATE, DELETE, LOGIN, LOGOUT)
- oldValues (JSON)
- newValues (JSON)
- createdAt (DateTime)

### Refresh Tokens Table
- id (UUID)
- userId (FK to Users)
- token (String, unique)
- expiresAt (DateTime)
- createdAt (DateTime)
